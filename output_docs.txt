TÀI LIỆU ĐẶC TẢ YÊU CẦU HỆ THỐNG
ĐỀ TÀI: HỆ THỐNG QUẢN LÝ BỆNH VIỆN (JAVA WEB SERVICE CUNG CẤP RESTful API)
I. Introduction (Giới thiệu)
1. Purpose (Mục đích)
Tài liệu Đặc tả Yêu cầu Hệ thống (SRS) này được biên soạn nhằm cung cấp một cái nhìn toàn diện và chi tiết nhất về dự án "Hệ thống Quản lý Bệnh viện". Mục đích của tài liệu là:
Đối với sinh viên (Nhóm phát triển): Cung cấp "bản vẽ kỹ thuật" định hướng rõ ràng về các tính năng cần lập trình, các ràng buộc kỹ thuật bắt buộc phải tuân thủ (RESTful, AOP, Spring Security) để tránh việc đi lạc đề hoặc thiếu sót tính năng.
Đối với giảng viên (Người đánh giá): Cung cấp bộ khung tiêu chuẩn để đối chiếu và chấm điểm dựa trên thang điểm 100 (60 điểm cho phần quản trị cơ bản và 40 điểm cho phần mở rộng nâng cao)
2. Application Overview (Tổng quan hệ thống) 
Hệ thống này được xây dựng dưới dạng ứng dụng Backend không trạng thái (Stateless Backend) cung cấp các RESTful API. Các ứng dụng phía Client (như Web ReactJS/VueJS hoặc Mobile App) sẽ tiêu thụ (consume) các API này để vận hành nghiệp vụ của một bệnh viện
Hệ thống đảm bảo các trụ cột chính:
Quản lý người dùng & Phân quyền: Xác thực chặt chẽ qua Spring Security và JWT.
Nghiệp vụ cốt lõi: Số hóa quy trình đặt lịch khám và quản lý hồ sơ bệnh nhân.
Giám sát hệ thống (Logging): Ứng dụng kỹ thuật AOP để tách biệt hoàn toàn code ghi nhật ký hệ thống ra khỏi code xử lý nghiệp vụ chính, giúp mã nguồn sạch và dễ bảo trì
3. Intended Audience and Reading Suggestions (Đối tượng và Hướng dẫn đọc) 
Backend Developer (Sinh viên): Cần đọc kỹ toàn bộ tài liệu, đặc biệt chú ý phần Use Case Specification để thiết kế Database và xây dựng các Endpoint (API) chính xác
Giảng viên/Người đánh giá: Có thể tập trung vào phần Security Requirement và Integration để đánh giá mức độ hoàn thành các tiêu chí nâng cao của dự án
4. Abbreviations (Thuật ngữ và viết tắt) 
API (Application Programming Interface): Phương thức giao tiếp giữa các phần mềm.
RESTful API: Chuẩn thiết kế API sử dụng các phương thức HTTP (GET, POST, PUT, DELETE) thao tác lên các Resource (Tài nguyên).
JWT (JSON Web Token): Tiêu chuẩn mã hóa chuỗi an toàn dùng để trao đổi thông tin người dùng (Token) giữa Client và Server.
AOP (Aspect-Oriented Programming): Lập trình hướng khía cạnh. Kỹ thuật giúp bắt các sự kiện (intercept) để xử lý logic phụ (như Logging) mà không can thiệp vào logic chính.
DTO (Data Transfer Object): Đối tượng dùng để đóng gói và truyền tải dữ liệu giữa Client và Server (giúp giấu đi cấu trúc Entity thực sự của Database)

III. Non-Functional Requirements
Performance (Hiệu năng):
Thời gian phản hồi API (API Response Time) trung bình phải < 2 giây
Giới hạn dung lượng tải lên file bệnh án (Upload file max size) là 10MB/file

Mật khẩu tuyệt đối không lưu Plain Text, phải mã hóa bằng thuật toán BCrypt với strength 10+

Maintainability (Khả năng bảo trì): Bắt buộc chia tách rõ ràng các Layer: Controller → Service → Repository
Logging: Bắt buộc ghi log các Request/Response quan trọng và exception qua AOP

V. Security Requirements
Bảo mật là yêu cầu cốt lõi. Sinh viên phải thiết lập cấu hình Stateless Session (không lưu Session trên Server) và sử dụng cơ chế bảo mật ba tầng Token như sau: 
1. Cơ chế hoạt động của Token (JWT) 
AccessToken: Token có tuổi thọ ngắn (VD: 15-30 phút). Được gửi kèm trong Header của mỗi HTTP Request để hệ thống nhận diện người dùng. Giúp hạn chế rủi ro nếu token bị lộ
RefreshToken: Token có tuổi thọ dài (VD: 7 ngày - 30 ngày). Khi AccessToken hết hạn, Client gửi RefreshToken đến API /api/auth/refresh để xin cấp lại một AccessToken mới mà người dùng không cần phải gõ lại Password
RevokedToken (Danh sách đen - Blacklist): Khi người dùng chủ động Đăng xuất (Logout) hoặc tài khoản bị Admin khóa, AccessToken hiện tại (dù còn hạn) phải được lưu vào một bảng TokenBlacklist trong Database. Bộ lọc (Filter) của Spring Security phải kiểm tra: Nếu Token nằm trong Blacklist -> Từ chối truy cập ngay lập tức

VII. Technical Conventions & Rules 
1. API Naming & Versioning Convention 
Versioning: Tất cả API phải có tiền tố version, bắt đầu bằng /api/v1/
Resource Naming: Sử dụng danh từ số nhiều (Plural nouns). Ví dụ: /api/v1/users, /api/v1/appointments
2. HTTP Status Code Standard 
200 OK: Truy vấn/Cập nhật thành công
201 Created: Tạo mới thành công (Đăng ký, Tạo lịch)
204 No Content: Xóa thành công
400 Bad Request: Lỗi Validation dữ liệu đầu vào
401 Unauthorized: Lỗi sai/thiếu Token
403 Forbidden: Không đủ quyền
409 Conflict: Xung đột dữ liệu (VD: Trùng lịch, Email đã tồn tại)
500 Internal Server Error: Lỗi hệ thống
